# Leadership Problem: Who Should Have Access to What

After 20+ years in consulting and MSP work, I've watched the same pattern kill permission cleanup projects across industries:

It's not bad tools. It's not lazy admins. It's one unanswered question:

**"Who should have access to what, and why?"**

## The Pattern

1. Copilot rollout exposes sensitive data
2. IT gets blamed for "bad permissions"
3. Remediation project kicks off
4. IT asks leadership: "What SHOULD access look like?"
5. Silence. Meetings. More silence.
6. Project stalls. Nothing changes.

→ Cleanup without governance is just expensive theater.

## Why This Question Gets Avoided

Defining access intent means:
- Named decision-makers (accountability)
- Documented risk acceptance (liability)
- Clear ownership of sensitive data (responsibility)

→ Nobody wants their signature on a decision that might look bad later.

The path of least resistance? Buy a tool, skip the hard conversations.

## The Questions That Stall Projects

- "Should HR data be accessible to department heads?"
- "Can contractors access the same files as employees?"
- "Who decides what's 'sensitive' in the first place?"
- "How long should project access last after completion?"

→ These aren't IT questions. They're business decisions IT gets asked to make alone.

## What Actually Works

The orgs that get this right treat access governance as a business decision, not an IT problem.

Before touching a single permission:

1. Get leadership to define access intent in writing
2. Assign data owners by department (not IT)
3. Document who accepts risk for each decision
4. Build governance FIRST, automate SECOND

→ Tools can enforce intent. They can't create it.

## The Bottom Line

Permissions aren't broken because IT failed. They're broken because nobody defined what "right" looks like.

Until someone answers "who should access what, for what purpose, for how long" - every remediation project is just rearranging deck chairs.
